Project Overview

This artifact is a C++ program originally developed as part of a Secure Coding course. The purpose of the program is to safely test numeric overflow and underflow across a wide range of C++ data types, including signed integers, unsigned integers, and floating-point types.

The original version of the program used template functions to repeatedly add or subtract values near the numeric limits of each data type. Before performing each operation, the code checked whether the next calculation would exceed the maximum or minimum value for that type. When a potential overflow or underflow was detected, the program printed a warning message to the console and stopped further processing for that test case.

This artifact demonstrates foundational secure coding concepts by proactively detecting unsafe numeric operations rather than allowing undefined or dangerous behavior to occur.

Code Analysis & Areas for Improvement

Key Issues Identified:

  • Tight Coupling: Overflow detection logic was directly coupled to console output through print statements
  • Error Handling: Functions simply printed messages and returned values instead of properly signaling errors
  • Separation of Concerns: Arithmetic logic and user output were tightly coupled and core functions both detected errors and printed messages
  • Testing Feedback: Test results required manual review of console output and there was no automated indication of success or failures
  • Code Structure: Calculation logic mixed with UI presentation concerns
  • Modern C++ Practices: Runtime type checks were used where compile-time checks could improve clarity and efficiency

Enhancements Implemented

1. Structured Error Handling with Custom Exceptions

A custom NumericOverflowException was introduced to replace console-only error reporting.

  • Overflow and underflow conditions now throw exceptions
  • Errors can no longer be silently ignored
  • Calling code can explicitly handle failure scenarios

This change improves reliability, security, and professional coding practices.

2. Separation of Logic and Output

Arithmetic functions were refactored to focus solely on computation and validation.

  • Core logic detects unsafe conditions and throws exceptions
  • Test functions are responsible for output and reporting
  • Improves modularity and reusability

3. Automated Test Result Tracking

Success and failure counters were added to the test runners.

  • Each test case records whether it passed or failed
  • Final summaries display clear test results
  • Reduces reliance on manual output inspection

4. Modern C++ Compile-Time Type Checking

Replaced runtime checks with if constexpr and std::is_integral_v<T>.

  • Type-specific logic resolved at compile time
  • Cleaner, safer, and more efficient code
  • Demonstrates modern C++ design principles

This enhancement showcases several key competencies in software design and engineering:

Narrative Reflection

The artifact I selected for this milestone is a program that I originally created during my Secure Coding course. The purpose of the original assignment was to explore how different C++ data types behave when pushed to their numeric limits. The program uses templates to test multiple primitive types and observes whether overflow or underflow occurs during repeated addition or subtraction. I chose this artifact because it represents foundational C++ concepts with templates, type traits, numeric limits, and highlights the importance of defensive programming, which is something I knew I could expand and refine for my ePortfolio.

I included this artifact because it gave me a strong opportunity to demonstrate software engineering and secure coding practices. The original version of the program worked, but it handled errors in a very informal way: it printed messages directly from the core logic and returned early whenever overflow was detected. To improve the quality of the program, I redesigned how the logic reports unsafe operations by creating a custom exception type, NumericOverflowException. Instead of printing messages inside the numeric functions, the program now throws structured exceptions that the test harness can catch and respond to. I also modernized the type-checking logic by replacing runtime if statements with C++17's if constexpr pattern, which allows compile-time branching based on whether a type is integral. Finally, I added automated pass/fail tracking so the program now produces a clear summary of how many operations succeeded and how many failed. These enhancements showcase my skills in clean code design, modern C++ practices, exception handling, and structured testing.

In Module One, I planned for this artifact to demonstrate primarily Outcome 3 (design and evaluate computing solutions) and Outcome 5 (develop a security mindset). I believe I fully met those goals. The new exception-based error handling shows a clear security mindset by ensuring that unsafe numeric behavior cannot be ignored by callers. The improved structure and separation of concerns—moving print statements out of the core logic—also demonstrates thoughtful software design and aligns strongly with Outcome 3. The addition of patterned test summaries also helps me better meet Outcome 2, because the output is now cleaner and easier for a user or reviewer to understand. Overall, the enhancements strengthened my alignment with multiple outcomes more than I originally anticipated.

Enhancing this artifact taught me a lot about how small architectural decisions can affect the clarity and safety of a program. I learned how powerful exception handling can be when used correctly, and how separating "logic" from "reporting" makes code easier to maintain and test. I also gained experience modernizing older C++ patterns with features like if constexpr, which helped me think more critically about compile-time vs. runtime decisions. One challenge I faced during the enhancement was balancing accuracy with readability, especially when handling floating-point overflow, which behaves differently than integral overflow. Another challenge was making sure the enhancements stayed true to the original purpose of the program, without over-engineering it. In the end, the process made me more confident in applying secure coding principles and taught me how to improve a program's testability without changing its core functionality.
